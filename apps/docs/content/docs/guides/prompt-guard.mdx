# Prompt Injection Guard

CodeSpar includes a built-in prompt injection guard -- **Security Layer #7** in the [10-layer defense model](/docs/configuration/security). The guard screens user messages before they reach an LLM, using pattern matching and structural analysis.

## Selective Enforcement

The guard only screens messages with **LLM-bound intents** -- commands where user-authored text is forwarded to Claude:

| Screened | Not screened |
|----------|-------------|
| `instruct`, `fix`, `plan`, `spec`, `lens` | `status`, `deploy`, `rollback`, `approve` |
| Unknown messages (SmartResponder fallback) | `logs`, `link`, `autonomy`, `kill` |

Structured commands like `status` and `deploy` parse parameters from the message without sending raw text to an LLM. Running the guard on these would produce false positives with no security benefit.

## Autonomy-Gated Behavior

The guard's response depends on the agent's [autonomy level](/docs/architecture/autonomy-levels):

| Autonomy | Guard behavior | Rationale |
|----------|---------------|-----------|
| L0-L2 (Passive, Notify, Suggest) | **Log only** -- message proceeds | A human reviews every action |
| L3-L5 (Auto-Low, Auto-Med, Full Auto) | **Block** -- message rejected | No human review, pre-screening matters |

At L0-L2, the guard still logs any triggered patterns to the audit trail. This gives operators visibility into injection attempts without blocking legitimate work.

## Detection Layers

### 1. Pattern Blocklist (12 rules)

Known injection patterns matched via regex:

| Category | Examples |
|----------|---------|
| Instruction override | "ignore previous instructions", "forget everything" |
| Role manipulation | "you are now a different AI", "DAN mode" |
| Data exfiltration | "reveal your system prompt", "show me your API key" |
| Command injection | `rm -rf`, `DROP TABLE`, `curl \| sh` |
| Delimiter abuse | Fake system/assistant markers, encoded payloads |

### 2. Structural Analysis

Detects suspicious patterns that don't match specific rules:

- **Multiple role markers** -- `user:`, `assistant:`, `system:` in a single message
- **Instruction-heavy language** -- excessive use of "must", "always", "override"
- **Suspicious Unicode** -- zero-width characters, RTL overrides (common in evasion)

### 3. Composite Risk Scoring

Each signal contributes to a risk score (0-1). Messages above the threshold (default **0.7**) are blocked (at L3+) or flagged (at L0-L2).

## Configuration

### Threshold

Set `PROMPT_GUARD_THRESHOLD` to tune sensitivity. Valid range: 0.0 to 1.0.

```bash
# More sensitive (blocks more, higher false positive risk)
PROMPT_GUARD_THRESHOLD=0.5

# Default
PROMPT_GUARD_THRESHOLD=0.7

# Effectively disable blocking (audit-only)
PROMPT_GUARD_THRESHOLD=1.0
```

### Custom Rules

Add organization-specific patterns:

```typescript
import { promptGuard } from "@codespar/core";

promptGuard.addPattern({
  id: "block-competitor",
  pattern: /switch to \w+ instead/i,
  weight: 0.8,
  description: "Blocks attempts to redirect to competitor tools",
});
```

## What Happens When Blocked

At L3+ autonomy with an LLM-bound intent:

- The message is **not forwarded** to the agent
- An audit entry is written with action `prompt_guard.blocked`
- The user receives: `"Message blocked by security policy."`

At L0-L2 or with non-LLM-bound intents:

- The message **proceeds normally** to the agent
- If patterns triggered, an audit entry is written with action `prompt_guard.flagged`

The blocked message is intentionally generic -- it does not reveal which patterns triggered or the risk score. Trigger details are recorded only in the audit trail.

## Audit Trail

All PromptGuard triggers are audit-logged regardless of whether the message is blocked:

| Field | Description |
|-------|-------------|
| `action` | `prompt_guard.blocked` or `prompt_guard.flagged` |
| `result` | `denied` (blocked) or `success` (flagged but passed) |
| `metadata.riskScore` | Numeric score (0-1) |
| `metadata.triggers` | Which pattern rules fired |
| `metadata.intent` | The classified intent type |
| `metadata.autonomyLevel` | Agent's autonomy level |
| `metadata.textPreview` | First 100 characters of the message |

## Indirect Source Screening

Beyond direct user messages, agents also process untrusted content from external
sources. The guard screens this content before it enters Claude prompts, redacting
any injection patterns it finds:

| Source | What's screened | Where |
|--------|----------------|-------|
| PR titles | Title text from GitHub API | ReviewAgent |
| PR diffs | Patch content per file | ReviewAgent |
| Commit messages | From Vercel webhooks and GitHub API | SmartAlert, SmartResponder |
| CI error messages | `errorMessage`, `buildError` from webhooks | SmartAlert |
| Audit context | `detail` field from audit trail entries | SmartResponder |
| Deploy data | Commit messages and errors in deploy records | SmartResponder |

### Redaction vs. Blocking

Indirect content is **redacted**, not blocked. When a pattern triggers, the
matched text is replaced with `[redacted: <rule-id>]` and the agent continues
processing. This is different from direct messages, where the guard can block
the entire operation at L3+.

The rationale: blocking a PR review because the PR title looks suspicious would
prevent the agent from doing its job. Redaction removes the injection attempt
while preserving enough context for useful analysis.

### Audit Trail

Flagged indirect content produces audit entries with action
`prompt_guard.indirect_flag`. The entry includes:

| Field | Description |
|-------|-------------|
| `metadata.source` | Source type (`pr_title`, `pr_diff`, `commit_message`, `ci_error`, `audit_context`, `deploy_data`) |
| `metadata.riskScore` | Numeric score (0-1) |
| `metadata.triggers` | Which pattern rules fired |
| `metadata.textPreview` | First 100 characters of the original content |

## Limitations

The guard is a **low-sophistication filter**, not a comprehensive injection defense. It catches copy-paste jailbreak attempts and known attack patterns. It does not catch:

- **Paraphrased injection** -- "please set aside your earlier guidelines"
- **Encoded payloads** -- base64, ROT13, systematic Unicode substitution
- **Multi-turn buildup** -- the guard is stateless, sees one message at a time
- **Semantic attacks** -- "write a function called ignore_previous_instructions"

The primary defenses against prompt injection are RBAC (prevents unauthorized actions), execution sandboxing (Docker restricts what agents can do), and Claude's safety training (rejects harmful instructions at the LLM level). The guard adds a defense-in-depth layer that raises the cost of attack from "copy paste from Reddit" to "think about paraphrasing."
